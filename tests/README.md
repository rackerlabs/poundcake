# Tests

Shell-based e2e tests in this directory exercise the PoundCake API workflow generation paths.

## Prerequisites

- Running PoundCake API endpoint
- `curl`
- `jq`
- Bash

## Run Tests By Hand

This is the canonical manual test path for shell e2e workflows in this repo.

### Prereqs Snapshot

- API is reachable (`compose` target defaults to `http://localhost:8000/api/v1`)
- `bash`, `curl`, and `jq` are installed
- `kubectl` is installed only when using `--target k8s`

### Quick Command Matrix

Compose full suite:

```bash
export POUNDCAKE_AUTH_SERVICE_TOKEN=shared-internal-key
./tests/run_all_e2e_workflow_generation_tests
./tests/run_single_task_non_blocking_test
./tests/run_multiple_tasks_with_blocking_test
./tests/run_reuse_recipe_two_webhooks_test
./tests/run_e2e.sh
```

Kubernetes full suite:

```bash
export POUNDCAKE_AUTH_SERVICE_TOKEN=shared-internal-key
./tests/run_e2e.sh --target k8s --namespace poundcake --service poundcake-api
```

Single runner:

```bash
./tests/run_e2e.sh --single run_single_task_non_blocking_test
```

List available runners:

```bash
./tests/run_e2e.sh --list
```

Debug run:

```bash
DEBUG=1 ./tests/run_e2e.sh --single run_single_task_manual_order_test
```

### How To Choose Commands

- Use full suite for confidence checks before merge/release.
- Use `--single` for fast local iteration.
- Use `--target k8s` when validating against a deployed cluster environment.

### Expected Pass/Fail Signals

- Success: command exits `0` and logs `PASS` and/or suite summary.
- Failure: command exits non-zero and logs the failing step label.

### Most Common Manual Mistakes

- Wrong k8s namespace/service for `--target k8s`.
- Local port collision in k8s mode when port-forward is enabled; set `--local-port` to an unused port.
- Proxy interference for cluster/localhost routes; set `API_URL` explicitly as needed.

## Environment Variables

Required for most workflows:

- `TEST_RECIPE` (auto-generated by run wrappers when omitted)

Common optional vars:

- `TEST_TARGET` (`compose` or `k8s`, default: `compose`)
- `API_URL` (default depends on target; remains explicit override)
- `POUNDCAKE_AUTH_SERVICE_TOKEN` (preferred when auth is enabled; sent as `X-Auth-Token`)
- `POUNDCAKE_SESSION_TOKEN` (optional session-cookie override for auth-enabled environments)
- `REQ_ID` (auto-generated by posting scripts when omitted)
- `IS_BLOCKING` (`true`/`false` for recipe creation scripts)
- `STEP1_CMD` (override step 1 command in two-step recipe creation script)
- `STEP2_CMD` (override step 2 command in two-step recipe creation script)
- `NON_BLOCKING_TOLERANCE_SEC` (allowed start-time delta for non-blocking near-simultaneous checks, default: `1`)
- `TEST_TIMEOUT_SEC` (default timeout used by helper library)
- `POLL_INTERVAL_SEC` (default poll interval used by helper library)
- `DEBUG=1` (enables helper debug logs)
- `POUNDCAKE_NAMESPACE` (default: `poundcake`, k8s mode)
- `POUNDCAKE_API_SERVICE` (default: `poundcake-api`, k8s mode)
- `POUNDCAKE_LOCAL_PORT` (default: `8000`, k8s mode local port-forward when enabled)
- `POUNDCAKE_REMOTE_PORT` (default: `8000`, k8s mode service port and direct URL port)
- `ENABLE_PORT_FORWARD` (`1` to enable k8s port-forward; default: `0`)

## API URL Resolution Precedence

`lib.sh` sets initial defaults first, then `run_e2e.sh` finalizes runtime `API_URL` after CLI parsing with this precedence:

1. `--api-url <url>` (always wins)
2. `--target k8s` with `--service <name>`:
   `http://<service>.<namespace>.svc.cluster.local:<remote-port>/api/v1`
3. Target default:
   - compose: `http://localhost:8000/api/v1`

## Execution Targets

### Compose mode (default)

- No automation needed.
- Default API endpoint: `http://localhost:8000/api/v1`

Example:

```bash
export POUNDCAKE_AUTH_SERVICE_TOKEN=shared-internal-key
./tests/run_e2e.sh
```

### Kubernetes mode (direct URL default)

- Run wrappers default to direct API URL access:
  - `http://<service>.<namespace>.svc.cluster.local:<remote-port>/api/v1`
- In `k8s` mode, provide either:
  - `--service <name>` (required unless `--api-url` is set), or
  - `--api-url <url>`
- Optional port-forward can be enabled explicitly:
  - `kubectl -n <namespace> port-forward svc/<service> <local>:<remote>`
- Defaults:
  - namespace `poundcake`
  - service `poundcake-api`
  - local:remote `8000:8000`
- `--local-port` is valid only with `--enable-port-forward`.
- Scripts auto-clean up the port-forward process on exit when enabled.

Example:

```bash
export POUNDCAKE_AUTH_SERVICE_TOKEN=shared-internal-key
./tests/run_e2e.sh --target k8s --namespace poundcake --service poundcake-api
```

Custom service/port example:

```bash
./tests/run_e2e.sh \
  --target k8s \
  --namespace my-ns \
  --service poundcake-api \
  --remote-port 8000
```

Port-forward example:

```bash
./tests/run_e2e.sh \
  --target k8s \
  --namespace my-ns \
  --service poundcake-api \
  --enable-port-forward \
  --local-port 18000 \
  --remote-port 8000
```

## Quick Start

Run full shell e2e workflow suite:

```bash
export POUNDCAKE_AUTH_SERVICE_TOKEN=shared-internal-key
./tests/run_e2e.sh
```

Run targeted workflows:

```bash
./tests/run_e2e.sh --single run_single_task_non_blocking_test
./tests/run_e2e.sh --single run_single_task_manual_order_test
./tests/run_e2e.sh --single run_single_task_webhook_counter_increment_test
./tests/run_e2e.sh --single run_multiple_tasks_non_blocking_test
./tests/run_e2e.sh --single run_multiple_tasks_with_blocking_test
./tests/run_e2e.sh --single run_reuse_recipe_two_webhooks_test
```

Notes:

- `run_single_task_manual_order_test` validates the direct `/orders` API path.
- `run_single_task_webhook_counter_increment_test` validates webhook counter increment semantics for repeated fingerprints.
- `run_multiple_tasks_with_blocking_test` asserts serial execution using dish task timestamps (`step_2.start >= step_1.end`).
- `run_multiple_tasks_non_blocking_test` asserts parallel behavior for two-step tasks (overlap or near-simultaneous starts).
- When API auth is enabled, shell e2e defaults to service-token auth via `POUNDCAKE_AUTH_SERVICE_TOKEN`.
- `POUNDCAKE_SESSION_TOKEN` is supported as an explicit override, but it is not the default e2e path.

## Blocking vs Non-Blocking Semantics

For the two-step workflow runners, blocking behavior is validated from task timestamps:

- Blocking recipe pass condition: step 2 starts at or after step 1 completes.
- Non-blocking recipe pass condition: step 2 starts before step 1 completes, or starts within `NON_BLOCKING_TOLERANCE_SEC` of step 1 start.

Assertion source is `dishes.result` when available, with fallback to `/dishes/{id}/ingredients` task records.

List available runner scripts:

```bash
./tests/run_e2e.sh --list
```

## Troubleshooting

- `API response is not valid JSON`:
  - API returned non-JSON text (often ingress/proxy error page or crash output).
  - Re-run with `DEBUG=1` and inspect printed response body.
- Squid / `ERR_INVALID_URL` / localhost proxy interception:
  - Use `--enable-port-forward` in k8s mode so wrappers set safe `NO_PROXY/no_proxy` values for localhost.
  - If setting `API_URL` manually, ensure it points to a reachable endpoint and bypasses proxies as needed.
- `API request failed ... (HTTP 4xx/5xx)`:
  - Endpoint rejected request; response body is printed by helpers.
  - Check recipe existence, auth headers, and API logs.
- `Auth appears to be enabled ... no e2e credentials were provided`:
  - Export `POUNDCAKE_AUTH_SERVICE_TOKEN` and rerun the suite.
  - Use `POUNDCAKE_SESSION_TOKEN` only when intentionally testing a session-authenticated path.
- Recipe already exists conflicts:
  - Use a unique `TEST_RECIPE` per run or allow wrappers to generate one.
- Timeouts waiting for orders/dishes:
  - Increase `TEST_TIMEOUT_SEC`.
  - Confirm workers and dependent services are healthy.
