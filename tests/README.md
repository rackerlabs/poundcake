# Tests

Shell-based e2e tests in this directory exercise the PoundCake API workflow generation paths.

## Prerequisites

- Running PoundCake API endpoint
- `curl`
- `jq`
- Bash

## Run Tests By Hand

This is the canonical manual test path for shell e2e workflows in this repo.

### Prereqs Snapshot

- API is reachable (`compose` target defaults to `http://localhost:8000/api/v1`)
- `bash`, `curl`, and `jq` are installed
- `kubectl` is installed only when using `--target k8s`

### Quick Command Matrix

Compose full suite:

```bash

./tests/run_e2e.sh
```

Kubernetes full suite:

```bash
./tests/run_e2e.sh --target k8s --namespace rackspace
```

Single runner:

```bash
./tests/run_e2e.sh --single run_single_task_non_blocking_test
```

List available runners:

```bash
./tests/run_e2e.sh --list
```

Debug run:

```bash
DEBUG=1 ./tests/run_e2e.sh --single run_single_task_manual_order_test
```

### How To Choose Commands

- Use full suite for confidence checks before merge/release.
- Use `--single` for fast local iteration.
- Use `--target k8s` when validating against a deployed cluster environment.

### Expected Pass/Fail Signals

- Success: command exits `0` and logs `PASS` and/or suite summary.
- Failure: command exits non-zero and logs the failing step label.

### Most Common Manual Mistakes

- Wrong k8s namespace/service for `--target k8s`.
- Local port collision in k8s mode; set `--local-port` to an unused port.
- Proxy interference on localhost; prefer `--target k8s` wrapper handling or set `API_URL` explicitly.

## Environment Variables

Required for most workflows:
- `TEST_RECIPE` (auto-generated by run wrappers when omitted)

Common optional vars:
- `TEST_TARGET` (`compose` or `k8s`, default: `compose`)
- `API_URL` (default depends on target; remains explicit override)
- `REQ_ID` (auto-generated by posting scripts when omitted)
- `IS_BLOCKING` (`true`/`false` for recipe creation scripts)
- `TEST_TIMEOUT_SEC` (default timeout used by helper library)
- `POLL_INTERVAL_SEC` (default poll interval used by helper library)
- `DEBUG=1` (enables helper debug logs)
- `POUNDCAKE_NAMESPACE` (default: `rackspace`, k8s mode)
- `POUNDCAKE_API_SERVICE` (default: `poundcake-api`, k8s mode)
- `POUNDCAKE_LOCAL_PORT` (default: `8000`, k8s mode local port-forward)
- `POUNDCAKE_REMOTE_PORT` (default: `8000`, k8s mode service port)

## Execution Targets

### Compose mode (default)

- No automation needed.
- Default API endpoint: `http://localhost:8000/api/v1`

Example:

```bash
./tests/run_e2e.sh
```

### Kubernetes mode (auto port-forward)

- Run wrappers auto-start:
  - `kubectl -n <namespace> port-forward svc/<service> <local>:<remote>`
- Defaults:
  - namespace `rackspace`
  - service `poundcake-api`
  - local:remote `8000:8000`
- Scripts auto-clean up the port-forward process on exit.

Example:

```bash
./tests/run_e2e.sh --target k8s --namespace rackspace
```

Custom service/port example:

```bash
./tests/run_e2e.sh \
  --target k8s \
  --namespace my-ns \
  --service poundcake-api \
  --local-port 18000 \
  --remote-port 8000
```

## Troubleshooting

- `API response is not valid JSON`:
  - API returned non-JSON text (often ingress/proxy error page or crash output).
  - Re-run with `DEBUG=1` and inspect printed response body.
- Squid / `ERR_INVALID_URL` / localhost proxy interception:
  - Use `TEST_TARGET=k8s` so wrappers set safe `NO_PROXY/no_proxy` values for localhost.
  - If setting `API_URL` manually, ensure it points to a reachable endpoint and bypasses proxies as needed.
- `API request failed ... (HTTP 4xx/5xx)`:
  - Endpoint rejected request; response body is printed by helpers.
  - Check recipe existence, auth headers, and API logs.
- Recipe already exists conflicts:
  - Use a unique `TEST_RECIPE` per run or allow wrappers to generate one.
- Timeouts waiting for orders/dishes:
  - Increase `TEST_TIMEOUT_SEC`.
  - Confirm workers and dependent services are healthy.
